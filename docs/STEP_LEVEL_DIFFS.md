## Step-Level Differences: Human Script vs Planner

This document compares the step-by-step flow in:
- `sentience-sdk-playground/amazon_shopping_with_assertions/main.py` (human-specified)
- `sentience-sdk-playground/planner_executor_local/main.py` (planner-generated)

It highlights where the planner flow diverges and why that reduces success rate.

## Human-Specified Flow (amazon_shopping_with_assertions)

The human script uses a fixed, deterministic sequence with explicit checks and fallbacks:

1. Navigate to Amazon homepage
2. Click search box (LLM selects input; strict instruction)
3. Type query with human-like jitter and verify search results URL
4. Click first product in results (explicit URL checks + content fallback)
5. Click Add to Cart (explicit text-based goal + drawer detection)
6. Navigate to cart page
7. Verify cart has items
8. Proceed to checkout
9. Verify checkout form elements and mark task complete

Key traits:
- Explicit waits and URL checks at each step
- Domain-specific guards for Amazon quirks
- Deterministic fallback logic for common failure modes
- Strict separation between navigation, selection, and verification

## Planner-Generated Flow (planner_executor_local)

The planner produces a compact plan with fewer, higher-level steps:

1. Navigate to Amazon homepage
2. Click search box
3. Type query and submit
4. Click first product link
5. Click Add to Cart (with optional substeps for drawer)
6. Proceed to checkout

Key traits:
- Step intents and verifies are generated by the planner
- Optional substeps rely on executor behavior and detection
- Fewer explicit “guard rails” for Amazon-specific flows

## Step-Level Differences

### Search box (Step 2)
- Human: LLM prompt explicitly forbids clicking links; retries on focus loss.
- Planner: Uses generic intent with less strict negative constraints.
- Impact: Planner can click unrelated links (language selector, menu items).

### Search results verification (Step 3)
- Human: Strong URL verification + fallback checks (URL patterns + not homepage/product).
- Planner: Relies on a single URL predicate, can miss redirects or intermediate pages.

### Product selection (Step 4)
- Human: URL + content-based fallback; avoids brittle selectors.
- Planner: Depends on model intent + executor selection; can drift to wrong link.

### Add to cart + drawer (Step 5)
- Human: Explicit “Add to Cart” instruction plus drawer detection logic.
- Planner: Optional substeps may be skipped or conflated with main click action.
- Impact: Executor can re-click Add to Cart instead of dismissing drawer.

### Cart navigation + validation (Steps 6–7)
- Human: Explicitly navigates to cart and verifies items.
- Planner: Skips cart verification, jumps to checkout intent.
- Impact: Less confirmation that cart is valid before checkout.

### Checkout (Steps 8–9)
- Human: Explicit step to verify checkout form elements; marks task done.
- Planner: One generic “proceed to checkout” click with URL predicate.
- Impact: Higher chance of mis-clicking or landing on a wrong page.

## Why Planner Success Rate Is Lower

1. **Model variability**: Planner intent and verify fields can drift or under-specify.
2. **Fewer guard rails**: Missing explicit cart verification and checkout validation.
3. **Executor dependency**: Wrong element selection amplifies planner drift.
4. **Optional substeps**: Drawer handling is best-effort, not guaranteed.
5. **UI volatility**: Amazon overlays and dynamic components need deterministic handling.

## Recommendations to Close the Gap

- Add explicit **cart verification step** after Add to Cart.
- Split **drawer dismissal** into a dedicated step when detected.
- Add stricter **search box constraints** (block language/settings links).
- Require **URL + content verification** for product page and checkout.
- Expand the planner template with **Amazon-specific guardrails**.
